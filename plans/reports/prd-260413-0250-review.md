## Next Sprint Candidates (Top 3 from Competitor Analysis)

### 1. Edit/Delete Bills — Effort: S
- **User story:** As a user, I want to edit or delete a bill I created so I can fix mistakes (wrong amount, wrong people) without recreating the whole expense.
- **Acceptance criteria:**
  - Bill creator (or group owner) sees Edit + Delete actions on bill detail / long-press in chat
  - Edit reopens BillConfirmSheet pre-filled; saving recomputes debts and replaces old debt rows
  - Delete prompts confirmation, soft-deletes bill (`deleted_at`), removes related debts, posts system chat message "Bill X đã bị xóa"
  - Telegram notify to affected users on edit/delete
- **Files to modify:** `src/components/bill/bill-confirm-sheet.tsx` (edit mode prop), `src/app/(app)/groups/[id]/page.tsx` (handleEditBill, handleDeleteBill, message render with actions), `src/lib/notifications.ts` (notifyBillEdited/Deleted), `src/app/api/notify/route.ts` (new types), supabase migration for `bills.deleted_at`
- **Why prioritized:** Both competitors have it, HIGH user value (mistake recovery is #1 support request), small surface area since BillConfirmSheet already exists. Unblocks many edge cases.

### 2. Simplify Debts Algorithm — Effort: M
- **User story:** As a group member, I want the app to minimize the number of transfers needed to settle, so instead of 6 separate payments I make 2.
- **Acceptance criteria:**
  - Group settings toggle "Tối giản nợ" (default ON for new groups)
  - When ON, debt banner + group debt summary uses simplified graph (A→B→C collapses to A→C)
  - Algorithm: net each person's balance, greedy match max-creditor with max-debtor until zero
  - Unit tests: 3-person cycle, 5-person chain, mixed positive/negative
  - Falls back gracefully if cycle: original per-bill debts still recorded for audit
- **Files to modify:** `src/lib/debt-calculator.ts` (new `simplifyDebts(balances)` fn), `src/app/(app)/groups/[id]/page.tsx` (use simplified for banner when toggle on), `src/app/(app)/groups/[id]/settings/page.tsx` (toggle UI), supabase migration for `groups.simplify_debts boolean default true`, `src/__tests__/debt-calculator.test.ts`
- **Why prioritized:** Splitwise's #1 differentiator. Pure-function algo, no UI rewrites. Directly reduces friction (fewer VietQR scans). Med effort = ~1 day.

### 3. Expense Categories — Effort: S
- **User story:** As a user, I want to tag bills with a category (Ăn uống, Di chuyển, Lưu trú, Khác) so I can later see where the group spends.
- **Acceptance criteria:**
  - 8 preset categories with emoji + Vietnamese label, hardcoded (no custom yet — YAGNI)
  - BillConfirmSheet shows horizontal scrollable category chips, default "Khác"
  - Category icon shows on bill row in chat + summary page
  - AI parser maps keywords ("ăn", "trưa", "taxi", "khách sạn") → suggested category
  - Stored as `bills.category text` (no FK, simple enum string)
- **Files to modify:** `src/components/bill/bill-confirm-sheet.tsx` (chip selector), `src/lib/categories.ts` (new — preset list + keyword map), `src/lib/ai-parser.ts` (extract category), `src/components/chat/chat-message-list.tsx` (render emoji), `src/app/(app)/summary/page.tsx` (group by category), supabase migration `bills.category`
- **Why prioritized:** Foundation for future analytics (sprint 4). Low effort, immediate UX value, plays to our AI parser advantage (auto-categorize is unique).

## Recommendation

**Implement #1 (Edit/Delete Bills) this cycle.**

Rationale: smallest effort, highest user-reported pain (no recovery from mistakes today), unblocks confidence in the AI parser (users will trust it more if mistakes are fixable). Categories (#3) is a close second but better paired with summary/analytics work in a later cycle. Simplify Debts (#2) is the highest *competitive* value but needs careful UX around the audit trail — defer to cycle 4 once edit/delete stabilizes the bill lifecycle.
